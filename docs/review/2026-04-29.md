# PentaVault Project Review - 2026-04-29

Completion Status: Complete - review snapshot recorded

On April 29, 2026, I conducted my first review of PentaVault. The project is still in an early development phase, so I approached it from the perspective of a new user in order to document the current user experience, identify areas for improvement, and capture the remaining work. Many features are still incomplete at this stage, but this review records the current behavior, working flows, and issues observed during testing.

This review focuses on user flow and the front-end experience. Code quality, implementation details, and system architecture will be reviewed later. The purpose of this document is to evaluate the UI, UX, and all front-end behavior, including any issues encountered by a normal user.

## Account Access and Authentication

Sign-in is currently working correctly. If a user does not have a valid email or ID, they cannot log in, which is expected. The "forgot password" flow also works as intended. If no email or password is associated with an account, the user can still enter an email address, but no email is sent because no account exists. This is good from a security perspective because it does not reveal whether an account exists.

I tested the email-sending backoff behavior. After sending three verification emails, the UI should show a wait period of approximately two to three minutes. The UI does display "Please try again after," and the feature is functioning as expected. Verification emails are also being received successfully.

I also tested an edge case where I partially completed the registration form, received the OTP, canceled the form, and then attempted to sign in. This behaved correctly. No database entries were created because the registration was not completed properly, which is the expected outcome.

I successfully created an account, entered the OTP, and received confirmation that the account was created correctly. Registration completed successfully.

During sign-in, I intentionally entered the wrong password and received an error stating that the ID and password may be incorrect. This is expected. The error does not specify which credential is incorrect, which is also good.

After signing in with the correct email and password, I reached the dashboard. My default personal organization was already created and assigned, which is expected and working well.

## Account Settings

After signing in, I updated my full name in the user-level account settings, and the update worked correctly.

I then tested password changes using the current password, new password, and confirm password fields. My first attempt used an incorrect current password, and the change failed as expected.

On the next attempt, I entered the correct current password and clicked "Change Password." I found a small flaw: if the current password and new password are the same, the password change is still allowed. If the current password and new password are identical, the user should receive an error. This rule should apply to all password-change flows, including OTP-based, MFA-based, and current-password-based changes. If the current password matches the new password, an error should be thrown. This should be addressed in a future update.

There is also a small bug after changing the password. The user is logged out, but this only becomes apparent after reloading the page or navigating to another page. This should also be fixed later.

Next, I tested changing the current password using email codes. I first checked whether the email-code flow followed the expected rate limits, exponential backoff, and retry behavior. The first issue I noticed is that after sending an email, the "Send again" button appears immediately. It should display a 60-second timer, and retries should only be allowed after that timer completes. This needs to be fixed. On the positive side, the underlying rate limiting appears to work properly.

One behavior is not necessarily a bug, but it is inconsistent. Changing the password with the current password prompts a re-login, while changing it with an email OTP does not log the user out, even though the user still needs to re-login. This behavior should be made consistent across both methods.

I covered most account-settings flows and edge cases, then moved on to MFA.

## Project Teams and Access

In the Access section of a project, under Teams and the Members list, the full list of members is shown. I want the current user's name to appear at the top with a "you" badge beside it, so users can quickly identify their own account.

The next bug I found is that when someone is promoted to admin of a project, they become an admin but also have the option to demote themselves to member. After doing so, they can lose the admin role, which is not intended.

As mentioned earlier, only the organization owner has ownership over all projects, so there is no owner role within a specific project. A project only has two roles:

- Member
- Admin

When a member is promoted to admin, that member can demote themselves back to member. This feels very odd and should be removed. Only the organization owner and project owner should have the right to change roles for everyone. This should be fixed.

The second bug is in Teams and access within a specific project. In Add members, there is an option to assign roles as developer, admin, or read-only. There is currently no developer role and no read-only project role. There should only be admin and member roles, so the developer and read-only options should be fixed.

Another small bug is that there is an option to join a project or add someone to a project, but there is no option for users to leave a project themselves or leave an organization themselves. This option should be added.

Admins can access all projects without being members of those projects, which is acceptable. When an admin creates a new project, they are automatically added to the project member list and cannot leave it. This should be fixed. Admins should have the option to move between projects. Every project is accessible to them, but they should be able to choose whether they are part of a project team. If they only want to create a project, they should be able to do that.

The same logic was considered for owners, but that part should be removed. An owner account is always added to every project. This is a security feature so that if access is lost through another account, all projects remain accessible through the owner account, similar to a root account. That behavior is correct.

The main requirement is to add the ability to leave a project. Admins should also have the option to be part of a project or not.

One important note: admins can promote other members to admin and demote other admins to member, but they cannot change their own role. An admin should not be able to update their own role to admin or member. This should be enforced.

When a member accesses the Teams page, they should be able to see all team members and their roles. Currently, they cannot. This should be enough for project-specific access and team roles for now.

## Organization-Level Access Control

I then returned to organization-level settings.

In Access Control, there are two options:

1. Members can see all projects.
2. Members can request project access.

I suggest removing the second option and making request capability part of the first option. When "Members can see all projects" is turned off, the second option does not make sense because users cannot request access to projects they cannot see.

When "Members can see all projects" is turned on, there should be an option to request access to a project if that capability is enabled.

The "Members can see all projects" option is not working properly. I tested it with one member. I turned off that member's project visibility, but projects were still visible to them. The request button becomes disabled when the option is deselected, but projects are still visible. This bug needs to be fixed.
